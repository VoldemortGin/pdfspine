# oxide-pdf PRD — Pre-Finalization Critique (applied)

*(The 25-item audit that the final PRD addresses. Kept for traceability.)*

This is a document audit task — critiquing a PRD. No skills match (the review/code-review skills target code diffs, not documents), and no files need to be read; the PRD is inline. I'll analyze it directly and return the prioritized fix list.

Below is my prioritized, numbered audit. Most important first.

---

**1. [§3.2 / §7 / §12 — CRITICAL SCOPE-VS-ROADMAP CONTRADICTION] "Rendering deferred" is contradicted by load-bearing v1 acceptance gates that require rendering.** Multiple v1 (M0–M5) exit gates depend on rasterization the PRD says is out of scope until M6:
- M4 exit (§12) requires "cross-viewer smoke (`pdftoppm`/pdfium) non-blank" — that is rendering annotations/forms to pixels.
- M5 exit requires `get_pixmap` on an image "== decoder output (pixel-equality)" — fine — but US-8 says "`get_pixmap` == decoder output," while §9.8 and §8.11 say vector pages raise `PdfUnsupportedError`. The boundary "image Pixmap in scope / vector Pixmap deferred" is asserted but never operationalized: there is no spec for what `page.get_pixmap()` does on a PDF page that contains *only* an image XObject (common: scanned PDFs). Is that "vector page" (deferred) or "image" (in scope)? This is the single most important ambiguity for the "LLM/RAG ingestion" headline use case (scanned docs). Define the exact rule.
- Fix: enumerate precisely which Pixmap paths are in v1, resolve the scanned-PDF case explicitly, and remove/replace render-dependent M4 acceptance signals with non-rendering checks.

**2. [§6.2 #8 / §6.4 / §10.6 — CLEAN-ROOM SELF-CONTRADICTION on the PyMuPDF oracle] The PRD simultaneously bans and uses PyMuPDF output.** §10.1 RED step says "paste red output in PR"; §10.6 + §6.2#8 permit PyMuPDF as an "ephemeral oracle behind `--features oracle-agpl`" whose "output is never stored as a golden." But US-2 ("fitz-compat suite green"), the "machine-checked compatibility matrix" (G7), and "serializer key/nesting/arity == PyMuPDF documented shape" (M2 exit) require knowing PyMuPDF's *exact* output shapes. If you may only use public docs, then any shape detail not in the docs (and PyMuPDF docs are notoriously incomplete on dict/rawdict field exactness) cannot be legally sourced. The "oracle that flags discrepancies for human investigation" *is* using AGPL output to drive expectations — a human reading the oracle diff and then changing code is exactly the contamination path clean-room is meant to prevent. This is the highest *legal* risk in the document and is currently hand-waved. Fix: state explicitly whether observed black-box *output values* (not just API names) may seed expectations, who is allowed to view oracle diffs (must be implementation-room-excluded personnel/agents), and how "human investigation" avoids becoming translation-by-osmosis.

**3. [§6.1 — LEGAL ERROR: black-box observation of AGPL software output is asserted as clean-room-safe; this is overstated.** "Feed PDFs into PyMuPDF, record input→output, reimplement to match observed behavior" is presented as unambiguously MAY. Running AGPL software to study its outputs is itself governed by the AGPL license you accepted to use it, and reproducing *expressive* output (e.g., exact HTML/XML serialization structure, specific error-message prose) can be derivative. Google v. Oracle covers API *declarations*, not output formats or behavior cloning. The PRD treats "behavioral compatibility" as categorically safe; it is not for creative/expressive outputs. Get this re-grounded by counsel and narrow the claim. The PRD lists this under "all verified industry behavior" without citation.

**4. [§6.3 / §6.4 — LICENSE POLICY INTERNAL INCONSISTENCY: LGPL/MPL.] §6.3 hard-blocks LGPL and MPL "in CI" and lists MPL as "❌ excluded by policy."** But:
- `hypothesis` (MPL-2.0) is a Python dev dep listed as ✅ in §6.4 — fine as dev-only, but the §6.3 matrix says MPL is CI-hard-blocked with no dev-only carve-out, so `cargo-deny`/policy text and the table contradict. State the dev-only/not-linked exception in the *policy*, not just the dep row.
- `insta` is repeatedly tagged "Apache" but is actually MIT/Apache-2.0 (and historically had an Apache-only period) — verify; minor but it's cited as a precise fact 3×.
- LGPL claim "relink obligation unsatisfiable for static Rust (no stable ABI/name mangling)" is the standard argument but is stated as fact; LGPL-3 §4 dynamic-relink can sometimes be satisfied. Soften or cite.

**5. [§8.4 / §8.10 / §3.2 — UNDERSPECIFIED HIGH-RISK: JBIG2/JPX have no real mitigation, only crate-swaps, and the in-scope/out-of-scope status is contradictory.** §3.2 is silent on JBIG2/JPX (implying in scope), §7 marks codecs P1/M5, §8.3 marks them "M5" decode, but the codecs are flagged "Very High" difficulty against admittedly "young permissive decoders" (R2/R3). The mitigation is entirely "use `hayro-jbig2`/`hayro-jpeg2000` + cross-check." Problems:
- `hayro-*` are pre-1.0 (0.3/0.4) and the PRD itself calls hayro "experimental" (R11). There is no fallback plan if a hayro codec cannot decode a real-world JBIG2 generic-region/symbol-dict file. "Cross-check vs oxidize-pdf" assumes oxidize-pdf's JBIG2 is correct — unestablished.
- No statement of *partial* JBIG2/JPX support being acceptable for v1, nor a defined behavior when decode fails (return raw? error? `PdfUnsupportedError`?). For the RAG persona, a scanned JBIG2 PDF that can't be decoded is a silent text-extraction failure, not a crash — that gap isn't captured by the "never panic" gate.
- Fix: define a degradation contract (decode-fail → typed error + extraction continues), make JBIG2/JPX explicitly P2-may-trail with a documented coverage subset, and name a concrete fallback (the OpenJPEG "optional fallback" is mentioned for JPX but is C — contradicts the pure-Rust/`#![forbid(unsafe_code)]`/WASM-clean selling points; flag that tension).

**6. [§9.6 / §9.1 — TECHNICAL TENSION: `#![forbid(unsafe_code)]` "everywhere except py-bindings" vs the actual dependency tree.** The core selling point is memory safety via forbidding unsafe. But `memmap2` (mmap is fundamentally unsafe — a truncated file under an active mmap is UB), `bytes`, `rustybuzz`, image codecs, and any optional OpenJPEG/pdfium fallback all contain or wrap `unsafe`. `#![forbid(unsafe_code)]` is a *crate-level* attribute that does not constrain dependencies. The PRD conflates "our first-party crates forbid unsafe" with "memory-safe engine" / "auditable supply chain (no opaque binary blob)" — mmap UB and codec unsafe are real attack surface. Fix: scope the safety claim precisely ("first-party Rust forbids unsafe; dependency unsafe is fuzzed + cargo-geiger-tracked"), and address the mmap-truncation UB explicitly (it's a known crash/UB vector for a malformed-input library whose #1 gate is "never panic/OOM").

**7. [§10 — TDD STATED BUT NOT OPERATIONALIZED at the gate that matters most.] The "tests-precede-implementation within a milestone" rule (`test-order-guard`, D-row, §10.1) is asserted but the mechanism is undefined and likely unenforceable as described.** A CI job cannot generally prove a test was authored before the implementation within a single squashed PR; "all milestone catalog entries become failing tests before any implementation merges" implies a milestone is *one giant red PR* merged before any green PR — which breaks D1 ("every catalogued test for the feature green") and D5 (diff-coverage ≥90%, "new public functions at 0% block merge") for that very PR. These two gates are mutually exclusive as written (you cannot merge a red test PR if 0%-coverage new code blocks merge). Resolve the test-first commit/merge workflow concretely (e.g., tests land `#[ignore]`-tagged with catalog status, or two-PR protocol), and reconcile D1/D5 with it.

**8. [§14 / §12 M2 / US-1 — UNFOUNDED ACCURACY CLAIM: "char accuracy ≥0.98 vs pdfminer.six" treats pdfminer as ground truth, which it isn't.** pdfminer.six and your engine will *both* be wrong on the same hard files (CID fonts w/o ToUnicode, broken encodings). Measuring Levenshtein *against pdfminer* means you inherit pdfminer's errors as your target and penalize yourself for being *more* correct. The metric needs a true ground-truth corpus (self-generated known-text PDFs, which §10.3 has) as the *primary* accuracy gate; pdfminer should be a sanity cross-check only. Same issue with "word-bbox IoU ≥0.90 vs pdfminer" — pdfminer's bboxes are character-level and don't define words the same way. Define the metric against ground truth and downgrade the pdfminer comparison.

**9. [§14 / R9 — PERFORMANCE CLAIMS partly implausible / internally soft.** "Within ~2× of pypdfium2 on cold open of large files" while using mmap + lazy load should plausibly *beat* pypdfium2 on cold open (you don't decode); meanwhile "text extraction within ~1.5–2× of pypdfium2" is optimistic for a from-scratch interpreter on a first release and has no basis. More importantly: there is no defined benchmark corpus, no hardware baseline, no methodology — so none of these KPIs are testable. "Far faster than pypdf" is the only safe claim. Fix: either drop the pypdfium2 ratios from v1 KPIs (R9 already says "not beat C everywhere") or attach a concrete bench harness + corpus + methodology so the numbers are gates, not vibes.

**10. [§3.2 / §7 — MISSING PyMuPDF CAPABILITIES not accounted for anywhere (neither in-scope nor explicitly non-goal).** An AI builder will hit these with no instruction:
- **Inline images (`BI/ID/EI`)** — §8.6 says "skip" for text extraction, but `get_images`/`extract_image`/Pixmap and redaction (image PIXELS mode) need real inline-image *decoding*, not skipping. Status undefined.
- **Tiling patterns / shading as fill in content streams** — even without rendering, text-on-pattern and redaction line-art modes touch these. Undefined.
- **`Page.get_drawings()` / `get_cdrawings()`** (vector path extraction) — heavily used by pdfplumber-style table/line analysis and by redaction "line-art modes" (§8.8). Not in §7 at all. Is it in scope? Redaction depends on it.
- **`Document.get_xml_metadata` / `set_xml_metadata`, `Page.get_label()`/page labels** — page labels are deferred (§3.2 #5) but `get_toc` with labels and named-dest "go to page N" interplay is unaddressed.
- **`Tools.mupdf_warnings`, `Tools.store_*`** — §3.2 #12 hand-waves "partial/by-design-different" — too vague to build.
- **`Widget` class API surface** (distinct from raw form-field dict access) — forms are P1/M4 but the `Widget` object (field_type, field_value, etc.) PyMuPDF API is not specified.
- **`Page.get_text("blocks", sort=...)` and the `flags` parameter defaults** — partially covered, but the *default* flag set (PyMuPDF's `TEXTFLAGS_*` defaults differ per method) is not pinned; shape parity requires exact defaults.
- Fix: add an explicit "deliberately-unlisted surface" subsection or expand §7 to cover get_drawings, inline images, Widget, and pin per-method default flags.

**11. [§8.7 / §3.2 #6 / US-5 — INCREMENTAL SAVE + SIGNATURE PRESERVATION claim is stronger than the spec supports.** US-5 / G3 promise "byte-exact incremental save" preserving signatures (`out[..orig.len()] == orig`). But §8.7 also says incremental save must "match original xref style" and the repair subsystem may have *rewritten* the in-memory model on open (Lenient mode, §8.2). If a file was opened via repair (reconstructed xref), `original_bytes` no longer correspond to a clean parse, and appending will not produce a validly-chained `/Prev`. The PRD never states that incremental save is **only** valid for files that parsed cleanly (no repair). For signed PDFs specifically, any prepended-junk header bias (§8.2 "header not at byte 0") breaks byte-offset assumptions. Fix: add precondition "incremental save requires a clean (non-repaired) parse; else reject/upgrade to full save," and reconcile with the signature-preservation promise.

**12. [§8.4 — ENCRYPTION TECHNICAL ERRORS / omissions.**
- The per-object key note "`MD5(filekey + objnum[3 LE] + gen[2] + "sAlT" for AES)`" is roughly right but the "+ 'sAlT'" 4-byte salt is appended *only* for AESV2, and the key length is `min(filekey_len + 5, 16)` — not stated; an AI builder will get the truncation wrong. 
- R5 (transitional AES-256) and R6 differ in the hash validation (R5 = single SHA-256; R6 = Algorithm 2.B iterated). The PRD lumps "R5 transitional" in but the *deprecation/security* note that R5 is broken/forgeable and should be read-but-not-written is missing (you say "write AES-256 R6" — good — but don't say "never write R5").
- AES-GCM (ISO 32003) is "tracked" — undefined what "tracked" means as a deliverable (in v1? no?). 
- Missing: behavior when `/Encrypt` references the document `/ID` but `/ID` is absent (common in malformed encrypted files) — KDF can't proceed; needs a defined fallback.

**13. [§9.2 / §9.4 / §9.7 — PyO3 MEMORY-MODEL claim needs scrutiny.** "Every `#[pyclass]` is `'static`, holds an `Arc`" + "`PyPage` carries its own `Arc` clone, not a `Py<PyDocument>` parent reference" is sound, but:
- §9.7 says `Document` is "serialized on write" via `RwLock`. If a `PyPage` holds an `Arc<DocumentStore>` and Python user code edits the document on thread A while reading the page on thread B, the PRD's "shared-immutable for parallel extraction, edits funneled through one thread" is a *documentation convention*, not an enforced invariant — yet the safety claim ("strictly safer than PyMuPDF") implies enforcement. With interior mutability + `Arc`, a read during a concurrent write is a `RwLock` contention/ordering issue, and any cached-but-now-stale derived data (e.g., a `TextPage` built from pre-edit content) is a correctness footgun. Clarify what is *enforced* vs *recommended*.
- Buffer protocol `unsafe` on `Pixmap` (§9.4) returns a view into Rust memory; if the `Pixmap`'s backing `Arc` can be dropped/mutated while a `memoryview`/numpy array is alive, that's a use-after-free. The PRD doesn't state the lifetime contract (PyO3 buffer protocol requires holding the export count). Specify it.

**14. [§8.8 — REDACTION: the destructive guarantee has a gap not covered by the byte-grep gate.** The gate is "whole-file byte-grep finds no secret + `get_text()` clean." But:
- Secrets in **image content** (scanned text under a redaction rect) are not caught by byte-grep of decoded text and are not addressed by "PIXELS blanks pixels" unless the redaction interpreter actually re-encodes the image region — which for JBIG2/JPX (transcoded on save, §8.3) means decode→blank→re-encode; that pipeline is unspecified and may not exist for those codecs in v1.
- Secrets in **font glyph names / ToUnicode / embedded font programs**, **document metadata/XMP**, **annotations you didn't author**, **JS / embedded files**, and **object streams not on the redacted page** survive byte-grep only if the grep is over the *fully decompressed* file — the gate says "whole-file byte-grep" which over a compressed save finds nothing regardless (false pass!). Specify: grep must run over fully-decompressed object data, and enumerate the surfaces redaction must scrub. This is a security-correctness bug in the acceptance gate itself.

**15. [§8.5 / §3.2 #9 — FONT SUBSETTING / `allsorts` license + scope inconsistency.** §3.2#9 + R5 + §7 say subsetting is feature-gated and "may trail," default = full embedding. But §14/US-4 success ("output stays small," "shared font single") and the merge dedup story lean on size; full-font embedding on `insert_text` with a CJK TTF bloats output massively — the "output stays small" KPI is at odds with "full embedding default." Also `allsorts` is Apache-2.0 (fine) but is described as "the only realistic subsetter" — single-point dependency for a P2 with no fallback; if it can't subset a given CFF/CID font, behavior undefined. Reconcile the size KPI with the no-subsetting-by-default decision.

**16. [§7 vs §3.2 — `convert_to_pdf` listed in TWO contradictory places.** §7 lists "Image documents … + `convert_to_pdf`" as **P0/M5**, but §3.2 #2 lists "`convert_to_pdf` for non-image inputs" as a **non-goal**, and #8 reiterates. The intent (image inputs → PDF is in scope; non-image → out) is decipherable but the §7 row says `convert_to_pdf` flatly as P0 without the qualifier, which an AI builder will read as "implement `convert_to_pdf` fully." Add the "(image inputs only)" qualifier in §7.

**17. [§12 — MILESTONE AWE / RISK MISALLOCATION is internally inconsistent with stated risk.** §8.2 says repair should "budget more than the whole writer," and R1/R16 call repair + M1 the risk concentration — yet M1 (parse+objects+filters+crypto-read+**repair**) is 12–16 AWE, *identical* to M3 (writer) and M4. If repair alone should exceed the writer, M1 must be substantially larger than M3, but they're equal. Either the "budget more than the writer" claim is rhetorical (then say so) or the AWE numbers are wrong. Also M2 (full text interpreter + font mapping + 10 serializers + search + layout) at 10–14 AWE is *less* than M1 and M3 despite arguably being the largest single subsystem — likely underestimated (R16 admits M2 risk but doesn't fund it).

**18. [§8.6 — TEXT coordinate/transform spec has a likely error.** "`Trm = params · Tm · CTM`" with row-vector PDF convention — PyMuPDF/PDF compose as text-space → Trm = params × Tm × CTM applied to row vectors, which is the order given, but the page-flip "`P = [1 0 0 -1 -mb.x0 mb.y1]`" must be applied *after* CTM (device transform), and the `/Rotate` adjustment is stated as "adjusted for /Rotate" with no formula. Rotation handling is the single most common text-extraction bug; "adjusted for /Rotate" is not actionable. Provide the four explicit page-transform matrices (0/90/180/270) including the MediaBox-origin translation, since shape-parity with PyMuPDF coordinates depends on getting this byte-for-byte.

**19. [§10.3 / §10.6 — CORPUS provenance risks under-mitigated.** "veraPDF + PDF Association corpora (permissive/CC, per-file checked)" — portions of the veraPDF corpus and PDF Association test files have *mixed/unclear* licensing and some are derived from copyrighted real documents; "per-file checked" is asserted but no process owner/tool is named beyond the manifest lint (which only catches `AGPL|GPL` strings + a hash blocklist — it cannot detect an unlicensed-but-not-AGPL file). GovDocs1 contains real scraped documents with unknown third-party copyright (it's used in research under fair-use norms, not a clean OSS license). For a project whose entire thesis is license cleanliness, the *fixture* corpus is a real exposure and the lint described cannot enforce the policy. Strengthen: define affirmative-license-required (not just AGPL-absent) and name who clears GovDocs1/veraPDF files.

**20. [§5.1 — COMPETITIVE TABLE factual nits that undercut credibility.** pypdf has had basic rendering removed long ago and never had raster render — "❌" is right, but pypdf *does* do some image extraction; "ReportLab license = BSD + commercial" is imprecise (ReportLab open-source is BSD-3, the commercial product is a separate paid library, not a dual-license of the same code — the framing implies dual-licensing like PyMuPDF, which is misleading); pikepdf is **MPL-2.0** but qpdf is **Apache-2.0** (the table says "Core: qpdf (C++)" with license MPL-2.0, conflating binding and engine — same conflation you correctly avoid elsewhere). Tighten so the licensing table (your core credibility asset) is exact.

**21. [§3.1 G1 / §14 — "≥99% of real-world PDFs" and "0 panics/OOM/hangs" lack a defined denominator.** ≥99% *of what corpus*? The number is only meaningful against a fixed, sized, named corpus snapshot. As written it's unfalsifiable. Pin the corpus + version + count, and define "open" (page_count correct? or just no-crash? — §14 says "correct page_count," §3.1 says "open and enumerate" — align).

**22. [§9.4 — abi3 + free-threading claim has a correctness caveat omitted.** abi3 (stable ABI) and the free-threaded (`abi3t`/PEP 703) build are *not* the same wheel and abi3 historically does **not** support the free-threaded ABI in a single artifact — you correctly ship them separately, but "one wheel per (OS,arch) covers CPython ≥3.10" is only true for the GIL builds; the text mixing abi3 and free-threading in one bullet invites the wrong impression. Also PyO3 0.29 + abi3 + free-threading support maturity should be version-pinned (Open Question 4 touches MSRV but not the abi3t/PyO3 support matrix). Minor wording + a pinned support claim.

**23. [§8.7 — GC level-3 "dedup identical objects … conservative on identity-bearing objects like page/annot" is underspecified and risky.** Structural-hash dedup that merges two structurally-identical but semantically-distinct objects (e.g., two annotations that happen to be identical, two pages sharing a Resources dict that later diverges) can corrupt files. "Conservative on identity-bearing objects" names the risk without defining the rule (which `/Type`s are excluded? how is reference-sharing-after-merge made safe under later edits given COW?). This is a known foot-gun (qpdf/PyMuPDF both restrict it); make the exclusion list explicit.

**24. [§15 Open Questions contains decisions that block M0/M1 and shouldn't be "open" in a build-driving PRD.** Q4 (MSRV), Q6 (Adobe AFM/AGL/CJK CMap licenses — these are *vendored data the clean-room thesis depends on*; "confirm before vendoring" means the font mapping layer in M2 is blocked on an unresolved legal question), Q9 (tolerance definitions — M2 exit gates literally cite "within tolerance" but the tolerance is an open question, so M2's gate is undefined), and Q10 (Limits defaults — M1's "never OOM" gate depends on them). A PRD that "drives an AI-assisted TDD build" cannot leave the *acceptance-gate numeric thresholds* (Q9) and *the legality of core vendored data* (Q6) open. Promote these to resolved requirements or the dependent milestone gates are non-actionable.

**25. [Cross-doc — MISSING SECTIONS for an "engineering-grade" PRD.** No section on: (a) **versioning/SemVer & API-stability policy** for the Rust crates and the Python package (matters because `fitz`-compat implies tracking a moving PyMuPDF surface); (b) **deprecation/compat policy** as PyMuPDF evolves (reference baseline is pinned to 1.24.14 — what happens at 1.25+?); (c) **observability/logging/diagnostics** for the warning collection (`Warning{offset,kind,detail}` is defined but how it's surfaced to Python users is unspecified); (d) **i18n/encoding of error messages**; (e) **build reproducibility / supply-chain (SLSA, cargo-vet, checksum pinning)** despite "auditable supply chain" being a headline claim — `cargo-deny` ≠ supply-chain attestation; (f) **maintenance/governance** (single-vendor risk on `hayro`, contribution model). At minimum add SemVer + the PyMuPDF-baseline-evolution policy; these directly affect the build.
